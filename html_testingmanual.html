Manual testing

cause You were born to test reality

Ep. 1: Vocation

User story:
As an IT enthusiast and someone who is super-dedicated to reforging his career, I want to know more about different aspects of IT, so that I can find the right place for my passion, interests, softskills' set and assets.

Acceptance criteria:
1. Learn HTML & CSS. Start building Your own page.
2. Start learning a programming language, e.g. JavaScript. Create some simple apps.
3. Find out more about databases and SQL. Download MySQL and create a database. Insert some values and try out some basic sql queries.
4. Realize that all Your experiences, all the actions and all the decisions You had made led to this precious moment: to the resolution of becoming a software tester.

Ep. 2: Theory

User story:
As a typical self-lerner and someone who is determined to become a software tester, I want to learn as much as possible, so that I can do really well during job interviews and be prepared to work.

Acceptance criteria:
1. Read some books on testing.
2. Get acquainted with the testing vocabulary.
3. Get acquainted with different types and levels of testing.
4. Find out that there is only on book You should really care about reading - the ISTQB syllabus.

Ep. 3: More theory - ISTQB

User story:
As a future QA Engineer I want to master the theory of testing, so that I can do truly understand every aspect of it.

Acceptance criteria:
1. Read the ISTQB syllabus carefully, from cover to cover.
2. Read it agan and again, until this fog of incomprehension finally disperses.
3. Sign up for the ISTQB Foundation Level exam to get a certificate.
4. Come to the conclusion that what You need now is getting help from a teacher who'd judge Your progress and show You how the testing is really done.

Ep. 4: Teachers

User story:
As a future QA Engineer I want somebody to evaluate my knowledge and show me how the testing process looks like, so that I can finally get a little bit closer to my destiny.

Acceptance criteria:
1. Find some free testing tutorials on Youtube.
2. Buy some testing courses on Udemy.
3. Learn how to create and manage testing documentation (test cases, bug reports etc.).
4. Start thinking that You're ready to move on with some real testing practice.

Ep. 5: Test cases

User story:
As a tester I want to learn some tools so that I am able to practice writing and executing test cases.

Acceptance criteria:
1. Find out what are the most popular testing tools nowadays.
2. Download them on Your local machine, install & and run.
3. Find some applications or websites, no matter real ones or dummy ones, and start testing.
4. Realize that what You're doing brings joy and satisfaction. Get ready to report those bugs You found.

Scenario: At first I examined the behaviour of a real online bookstore searchbox. With no specification at hand all I could do was to validate whether the searchbox behaviour met my personal expectations. In other words: I made this test fail on purpose hoping the online bookstore owners won't ever hear about this.

Environment:
Tool: Testlink (v. 1_0_20_fixed)
OS: Windows 10 Home 64-bit
Browser: Google chrome

Ep. 6: Bug reports

User story:
As a tester I want to learn some tools so that I am able to practice writing bug reports.

Acceptance criteria:
1. Find out what are the most popular bug-reporting & issue-tracking tools nowadays.
2. Download them on Your local machine, install & run.
3. Find some applications or websites and start testing them. You can use either real ones or dummy ones (filled with bugs on purpose).
4. Come to the realization that You're ready to apply for the job advertisement You found interesting. Send Your résumé.

Scenario: The mobile chat application bug report I wrote was part of a recruitment process run by a certain company. I downloaded the app from a link sent via e-mail, installed it on my mobile and tested through. Whether I passed or failed will remain my secret :)

Environment:
Bugtracker: ms word document
Tested app: LeanCode Chat
Device 1 / RAM / OS: Xiaomi Redmi Note 8T / 4 GB RAM / Android v. 11;
Device 2 / RAM / OS: Samsung Galaxy A21s / 3 GB RAM / Android v. 12;

PDF BUG REPORT

Scenario: This time I decided not to make up bugs myself, but to use the ones created by someone else for me to find them and report them. I played with Mr Buggy 6 for a while and decided the login form best suits my needs at this point. The case was simple: it was possible to establish a password different than the password-rules provided.

Environment:
Bugtracker: Jira
Tested app: Mr Buggy 6
OS: Windows 8.1
